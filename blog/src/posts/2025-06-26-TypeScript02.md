## 인터페이스

타입스크립트에서 객체 타입을 정의할 때 사용하는 문법

- 객체 속성/속성 타입
- 함수의 파라미터, 반환 타입, 스펙(파라미터 개수와 반환값 여부 등)
- 배열과 객체를 접근하는 방식
- 클래스

```tsx
// interface 이름은 파스칼케이스로 작성
interface Person {
  name: string;
  age: number;
}

function getPerson(someone: Person): Person {
  return someone;
}
```

### 인터페이스 옵션

한 속성은 없어도 가능하게끔 만들고 싶을 때 ? 를 붙여서 사용

```tsx
interface Person {
  name?: string;
  age: number;
}
```

- 옵셔널 속성을 남발하면 안 좋은 패턴임!
  ```jsx
  // 나쁜 예
  interface User {
    id: number;
    name?: string;
    email?: string;
    age?: string;
  }
  ```

  - 쓸 때마다 `undefined` 체크를 강제함
  - 어떤 필드가 실제로 필요한지 의도가 불명확해짐
  - 타입이 주는 "보장"이 약해짐
  ```jsx
  // 상황에 맞게 타입을 분리 (좋은 예)
  interface GuestUser {
    id: number;
  }

  interface RegisteredUser {
    id: number;
    name: string;
    email: string;
  }

  type User = GuestUser | RegisteredUser;
  ```

### 인터페이스 상속

JS class 상속과 동일하게 extends 예약어 사용

자식은 부모에 정의된 타입을 수정할 수 없음 ⇒ 객체지향의 **리스코프 치환 원칙(LSP)**

```tsx
interface Person {
  name: string;
  age: number;
}

interface Developer extends Person {
  skill: string;
}

var ironman: Developer = {
  name: "아이언맨",
  age: 41,
  skill: "만들기",
};
```

### 인덱스 시그니처 (인터페이스에서 인덱싱 타입 정의)

동적 키를 가진 객체의 타입을 정의할 때 사용

즉, 키로 뭐가 들어올지 모를 때 타입을 미리 정의해두기 위해서 사용함

```tsx
// 키를 다 알면 그냥 이렇게
interface User {
  name: string;
  age: number;
}

// 키를 모를 때 (동적으로 추가되는 경우)
interface Dict {
  [key: string]: string;
}
```

객체의 property name과 개수가 정해져있으면 명시하는 것이 좋음 → 자동완성 기능 사용 가능

```tsx
interface User {
  [property: string]: string;
  id: string;
  name: string;
}

var seho: User = {
  id: "1",
  name: "세호",
  address: "판교", // 인덱스 시그니처로 인해 새로운 키-밸류 추가 가능
};
```

중요한 점!!

인덱스 시그니처는 **"이 객체 안의 모든 키에 적용되는 규칙"** 이라서 미리 정의한 속성도 그 규칙을 벗어날 수 없음

```tsx
interface User {
  [property: string]: string;
  // "모든 string 키 → 값은 string"
  age: number; // ❌ age의 값이 number → 인덱스 시그니처 규칙 위반
}

// 만약 age: number도 허용하고 싶다면?
// Union 타입으로 처리해야 함
interface User {
  [property: string]: string | number; // number도 허용
  id: string;
  name: string;
  age: number; // ✅ OK
}
```

### 선언 병합(Declaration Merge)

같은 이름으로 선언한 interface는 자동으로 병합됨

```tsx
interface User {
  id: string;
  name: string;
}

interface User {
  // 같은 이름으로 또 선언
  age: number;
}

// 자동으로 합쳐짐
const user: User = { id: "1", name: "Kim", age: 25 }; // ✅
```

실무에서는 라이브러리 타입 확장할 때 자주 사용됨

## 유니언 타입

여러개의 타입 중 한 개만 쓰고 싶을 때 사용하는 문법

타입에 따른 메소드들을 자동 완성으로 사용 가능하고 any보다 명시적임

```tsx
function logText(text: string | number): void {
  console.log(text);
}
```

### 유니언 타입 사용 시 공통 속성이 아닐 때 주의

아래 예제에서 Person과 Developer가 가지고 있는 속성이 다를 때, 공통 속성만 출력 가능

이럴 때에는 in 연산자로 분기처리

in 연산자는 객체에 해당 속성이 있으면 true, 없으면 false 반환

```tsx
interface Person {
  name: string;
  age: number;
}

interface Developer {
  name: string;
  skill: string;
}

function introduce(someone: Person | Developer): void {
  // console.log(someone.age); // 오류! Developer일 때는 age 없음
  if ("age" in someone) {
    console.log(someone.age);
    return;
  }
  if ("skills" in someone) {
    console.log(someone.skill);
    return;
  }
}
```

## 인터섹션 타입

타입 2개를 & 로 합쳐서 사용 ⇒ 두 가지 속성을 모두 만족하는 객체를 넘겨야 함

```tsx
interface Avenger {
  name: string;
}

interface Hero {
  skill: string;
}

function introduce(someone: Avenger & Hero) {
  // 인터섹션 타입이므로 둘 다 출력 가능
  console.log(someone.name);
  console.log(someone.skill);
}

introduce({ name: "캡틴", skill: "어셈블" }); // Avenger, Hero 모두 충족시키므로 오류 X
introduce({ name: "그루트" }); // skill이 없으므로 오류 O
```

실무에서 자주 쓰는 패턴은 **기존 타입에 속성을 추가할 때**

```tsx
interface User {
  id: string;
  name: string;
}

// User에 권한 정보만 추가하고 싶을 때
type AdminUser = User & { role: "admin" };
```

남발하면 타입이 복잡해져서 가독성이 떨어짐

이럴 때는 차라리 extends를 사용
